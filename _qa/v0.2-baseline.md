Round-trip covers:

- write of the ENQUEUE line

## Architecture delivered

```
client (one-shot)              daemon (long-lived)
  argv → Message                accept loop
  connect UNIX socket  ───────► Stream
  write ENQUEUE\t...   ───────► parse + push Queue
                                signal cond
  read OK\t<id>        ◄─────── write OK
  exit                          ▼
                                worker thread
                                pop (cond wait)
                                spawn say + wait
                                loop
```

The worker thread serializes playback — never two `say` processes in parallel (UX requirement).
In-memory queue `std.ArrayList(Message)` + `std.Io.Mutex` + `std.Io.Condition`.

## v0.2 → v0.3 gaps

- In-memory queue: does not survive restart (v0.3 = SQLite WAL)
- No `agent-tts queue` / `skip` / `clear` (v0.3)
- Daemon foreground-only — user has to start it manually (v0.4 = launchd)
- Pre-warm happens once at boot, but every `say` still does a fresh fork+exec. In v1.0+ evaluate embedding `AVSpeechSynthesizer` if real TTFA measured (dtruss) lands above target
- Real TTFA (audio-sample detection) still not measured — only round-trip ACK + spawn proxy
