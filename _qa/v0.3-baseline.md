## Persistence (smoke test)

Scenario:

1. Daemon clean, queue empty.
2. Enqueue 3 items (rate 100, long text).
3. `agent-tts queue` shows `id=8 playing, id=9 pending, id=10 pending`.
4. `kill -9` the daemon and the `say` process during item 8's playback.
5. DB inspection: `8|playing, 9|pending, 10|pending` (item 8 orphaned).
6. `agent-tts daemon` restart.
7. Daemon log: `[daemon] recovered 3 pending items from previous run`.
8. Wait for drain.
9. Final DB: `8|done, 9|done, 10|done`.

**Result**: queue survives crash. Orphaned `playing` item is re-promoted to `pending` at boot (`UPDATE state='pending' WHERE state='playing'` in `Queue.init`). Worker drains sequentially with no loss.

## Worker crash recovery

When the daemon dies during `say`:
- The `say` child becomes orphaned (subprocesses don't die with a kill -9 on the parent — confirmed via pgrep in the test).
- The item's row stays at `state='playing'` in SQLite. Without cleanup, it would stay there forever.
- `Queue.init` runs `UPDATE items SET state='pending', started_at=NULL WHERE state='playing'` at boot. Idempotent.

Known trade-off: the recovered item gets **spoken again from the beginning** by the new daemon. There is no audio-offset checkpoint (you can't resume `say` mid-stream). Acceptable for v0.3 — better UX than losing the item.

## SQLite operations

- `PRAGMA journal_mode=WAL` enabled at boot.
- `PRAGMA synchronous=NORMAL` — at worst loses the last non-fsynced transaction on an OS crash (not on a process kill).
- Partial index `WHERE state IN ('pending','playing')` keeps the active queue small even with thousands of historical items.
- Single connection shared between accept thread and worker thread, serialized via `std.Io.Mutex` around each operation.

## IPC protocol v0.3

Text line, tab-separated. Same as v0.2 plus 3 new ops:

```
Client → daemon:
  ENQUEUE\t<voice>\t<rate>\t<text>\n
  QUEUE\n
  SKIP\n
  CLEAR\n

Daemon → client:
  OK\t<id>\n                                          (ENQUEUE / SKIP / CLEAR)
  ITEM\t<id>\t<state>\t<voice>\t<rate>\t<text>\n     (QUEUE, one per item)
  END\n                                                (QUEUE, end of list)
  ERR\t<msg>\n                                         (any op)
```

Backward compatible: `agent-tts "text"` still works (same ENQUEUE op).

## Gotchas resolved in v0.3

- `std.Io.File.stdout().writer(io, buf)` defaults to `positional` mode, which silently fails on pipes/ttys — the write appears OK but nothing arrives. Fix: `writerStreaming(io, buf)`. Confirmed: the first call works, subsequent calls swallow output with no error.
- `Reader.takeDelimiterExclusive('\n')` does NOT consume the delimiter — only what came before it. A naive loop generates empty reads forever. Fix: `takeDelimiterInclusive` + strip the `\n`.
- `SQLITE_TRANSIENT` (`-1` cast to a pointer) trips `@ptrFromInt` on alignment. Workaround: `SQLITE_STATIC` (= null) — valid because the bind→step→finalize cycle is synchronous and buffers live in the arena.
- `@cImport(sqlite3.h)` resolves via the macOS SDK sysroot without any extra flag. `linkSystemLibrary("sqlite3", .{})` on the module + `link_libc = true` is enough.
