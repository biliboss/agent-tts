## Persistência (smoke test)

Cenário:

1. Daemon limpo, fila vazia.
2. Enqueue 3 items (rate 100, texto comprido).
3. `agent-tts queue` mostra `id=8 playing, id=9 pending, id=10 pending`.
4. `kill -9` no daemon e no `say` durante a fala do item 8.
5. Inspeção DB: `8|playing, 9|pending, 10|pending` (item 8 orfão).
6. `agent-tts daemon` restart.
7. Log do daemon: `[daemon] recovered 3 pending items from previous run`.
8. Aguardar drain.
9. DB final: `8|done, 9|done, 10|done`.

**Resultado**: fila sobrevive crash. Item órfão em `playing` é re-promovido pra `pending` no boot (`UPDATE state='pending' WHERE state='playing'` em `Queue.init`). Worker dreina sequencialmente sem perda.

## Recuperação de crash do worker

Quando o daemon morre durante `say`:
- Filho `say` vira órfão (subprocesses não morrem com kill -9 do parent — confirmado via pgrep no teste).
- Linha do item fica em `state='playing'` no SQLite. Sem cleanup, ficaria pra sempre.
- `Queue.init` faz `UPDATE items SET state='pending', started_at=NULL WHERE state='playing'` no boot. Idempotente.

Trade-off conhecido: o item recuperado vai ser **falado de novo do começo** pelo novo daemon. Não há checkpoint de offset de áudio (não dá pra retomar `say` no meio). Aceitável para v0.3 — UX melhor que perder o item.

## Operações SQLite

- `PRAGMA journal_mode=WAL` ativado no boot.
- `PRAGMA synchronous=NORMAL` — perde no máximo a última transação não-fsynced em crash do OS (não no kill do processo).
- Index parcial `WHERE state IN ('pending','playing')` mantém a fila ativa pequena mesmo com milhares de itens históricos.
- Connection única compartilhada entre accept thread e worker thread, serializada via `std.Io.Mutex` ao redor de cada operação.

## Protocolo IPC v0.3

Linha texto, tab-separado. Igual v0.2 + 3 novas ops:

```
Client → daemon:
  ENQUEUE\t<voice>\t<rate>\t<text>\n
  QUEUE\n
  SKIP\n
  CLEAR\n

Daemon → client:
  OK\t<id>\n                                          (ENQUEUE / SKIP / CLEAR)
  ITEM\t<id>\t<state>\t<voice>\t<rate>\t<text>\n     (QUEUE, one per item)
  END\n                                                (QUEUE, end of list)
  ERR\t<msg>\n                                         (any op)
```

Compatibilidade: `agent-tts "texto"` continua igual (mesma op ENQUEUE).

## Gotchas resolvidas em v0.3

- `std.Io.File.stdout().writer(io, buf)` usa modo `positional` por default, que falha silenciosamente em pipes/ttys — escrita parece OK mas nada chega. Fix: `writerStreaming(io, buf)`. Confirmado: primeira chamada funciona, segundas em diante comem o output sem erro.
- `Reader.takeDelimiterExclusive('\n')` NÃO consome o delimitador — só o que veio antes. Loop simples gera reads vazios infinitamente. Fix: `takeDelimiterInclusive` + strip do `\n`.
- `SQLITE_TRANSIENT` (`-1` cast pra ponteiro) trava `@ptrFromInt` por alinhamento. Workaround: `SQLITE_STATIC` (= null) — válido porque o ciclo bind→step→finalize é síncrono e os buffers vivem na arena.
- `@cImport(sqlite3.h)` resolve pelo sysroot do SDK macOS sem flag extra. `linkSystemLibrary("sqlite3", .{})` no module + `link_libc = true` basta.
