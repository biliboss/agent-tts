# v0.4 — launchd auto-start (Mac Air M4)

Date: 2026-06-03
Binary: `zig-out/bin/agent-tts` (Zig 0.16, ReleaseFast, 481KB / 492584 bytes, arm64 Mach-O)
Plist label: `io.github.biliboss.agent-tts` (default) — `AGENT_TTS_LAUNCHD_LABEL` env override used in this test
Daemon: launchd LaunchAgent (RunAtLoad + KeepAlive on non-zero exit)

## Dry-run round-trip (test label, does not touch real install)

The environment variable isolates the test from the permanent install:

```bash
export AGENT_TTS_LAUNCHD_LABEL=io.github.biliboss.agent-tts.test
./zig-out/bin/agent-tts daemon install
./zig-out/bin/agent-tts daemon status
./zig-out/bin/agent-tts daemon uninstall
```

3 round-trips measured with `/usr/bin/time -p`:

| Run | install (s) | uninstall (s) |
|-----|-------------|---------------|
| 1   | 0.01        | 0.01          |
| 2   | 0.01        | 0.01          |
| 3   | 0.01        | 0.01          |

**Median: ~10ms install + ~10ms uninstall**. Dominated by `/bin/launchctl` fork+exec. `/usr/bin/time` granularity on macOS is 10ms — real ≤ 10ms (probably ~5ms).

Covers:
- `std.process.executablePath` (resolves absolute path via `_NSGetExecutablePath` on Darwin)
- `createFileAtomic` + `replace` in `~/Library/LaunchAgents/`
- `launchctl bootstrap gui/<uid> <plist>` (install) — modernized to avoid the deprecated `launchctl load`
- `launchctl bootout gui/<uid> <plist>` (uninstall)
- `launchctl print gui/<uid>/<label>` (status, parsing-free passthrough)

## Refusal paths (defensive guards)

| Scenario | Result |
|---------|-----------|
| `daemon install` when the plist already exists | error "already installed at <path> — uninstall first", exit 1 |
| `daemon uninstall` when the plist does not exist | error "not installed (no plist at <path>)", exit 1 |

Both install and uninstall refuse the wrong state, avoiding double-bootstrap (which launchctl rejects with EALREADY) and bootout against a ghost label.

## Plist contents (redacted)

`plutil -lint <plist>` → `OK`. Size: 1099 bytes (real host). Contents (HOME redacted):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>io.github.biliboss.agent-tts</string>
    <key>ProgramArguments</key>
    <array>
        <string>$HOME/.../agent-tts/zig-out/bin/agent-tts</string>
        <string>daemon</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <dict>
        <key>SuccessfulExit</key>
        <false/>
    </dict>
    <key>StandardOutPath</key>
    <string>$HOME/.cache/agent-tts/daemon.out.log</string>
    <key>StandardErrorPath</key>
    <string>$HOME/.cache/agent-tts/daemon.err.log</string>
    <key>WorkingDirectory</key>
    <string>$HOME/.cache/agent-tts</string>
    <key>EnvironmentVariables</key>
    <dict>
        <key>HOME</key>
        <string>$HOME</string>
    </dict>
    <key>ProcessType</key>
    <string>Background</string>
</dict>
</plist>
```

Key decisions:

- **KeepAlive as a dict with `SuccessfulExit=false`**: a plain `<true/>` restarts even after a clean shutdown — `bootout` would be chasing its own tail. `SuccessfulExit=false` lets exit 0 be respected; restart only on crash.
- **HOME in EnvironmentVariables**: launchd does not reliably inherit `HOME` across all contexts (pre-login, ssh, etc). Without it, `ipc.socketPath` resolves to `/tmp/.cache/agent-tts/sock` and the client cannot find the daemon.
- **WorkingDirectory = `~/.cache/agent-tts`**: ensures relative log paths (if the daemon ever writes without an absolute path) land in the right place.
- **ProcessType = Background**: a hint to the macOS scheduler — lower I/O priority, no application window.

## Implementation notes (Zig 0.16)

- **Self-locate**: `std.process.executablePath(io, &buf)`. On Darwin the vtable uses `_NSGetExecutablePath` + realpath (see `std.Io.Threaded.processExecutablePath`). Argv[0] was rejected: it arrives as `agent-tts` when invoked via PATH and launchd refuses relative paths in `ProgramArguments`.
- **uid lookup**: `std.c.getuid()` (direct FFI to libc). `std.posix` does not expose getuid as a public function in 0.16 — `std.c` is the standard channel. Domain string: `gui/<uid>`.
- **Atomic write**: `Dir.createFileAtomic` creates an ephemeral file inside `~/Library/LaunchAgents/` (same filesystem → atomic rename), `Atomic.replace` does the swap. launchd sees either the old or the new plist — never half-written.
- **Plist existence check**: `Io.Dir.accessAbsolute(.read = true)` (note: free function, not a method in 0.16 — `cwd().accessAbsolute(...)` does not exist).
- **XML escape**: 5 entities (`& < > " '`). Necessary because some Macs have HOME paths with spaces, and Apple accepts `&` in paths (rare, but the `xmlEscape` test covers it).

## v0.4 → v0.5 gaps

- A daemon already running when install fires conflicts on the socket — install does not detect or kill the old daemon. Current workaround: `pkill -f "agent-tts daemon"` before install. v0.5 may add an "uninstall pre-clean".
- No `daemon restart` / `daemon reload` — use `uninstall && install`.
- No `daemon logs` (tail of the .log files). v0.5 or v0.6 depending on pressure.
- Pre-warm `say -v Luciana " "` still runs on every launchd (re)start — good (warm at boot), but on a laptop with aggressive sleep, launchd may kill the daemon and re-spawn it many times per day. KeepAlive covers it, but costs 340ms × N. Monitor.

## v0.4 → SCQA

**S** (situation): v0.2 delivers a foreground daemon; the user has to remember to start it manually each session. Logout/reboot kills everything.
**C** (complication): the KPI is TTFA. The first call of the day pays cold start whenever the user forgot `agent-tts daemon` in a terminal. Which is every time.
**Q** (question): How do we keep the daemon up without the user thinking about it?
**A** (answer): LaunchAgent plist in `~/Library/LaunchAgents/` + `launchctl bootstrap gui/<uid>` at install. Mac boot, login, sleep/wake — launchd resurrects the daemon, Luciana already pre-warmed. The warm TTFA preserves the 0ms round-trip from v0.2.
