# v1.2 baseline — streaming · 2026-06-03

Mac Air M4, ReleaseFast, `-Dwith-piper=true`. Faber-medium voice (22050 Hz). Daemon piper engine resident. Bench command:

```bash
agent-tts ttfa-bench --engine piper --warm 1 --input long
```

Long input: `_qa/v1.2-long-input.txt` — 490 Pt-BR words, 3085 bytes, 47 sentence chunks after `preproc.chunkSentences`.

## Streaming pipeline (v1.2)

| Metric | Run 1 | Run 2 |
|---|---:|---:|

| Long input total wall time | 166 647.8 ms | 166 284.2 ms |
| Inter-chunk gap median | 0.02 ms | 0.02 ms |
| Inter-chunk gap max | 0.16 ms | 0.61 ms |

## What the numbers mean

- **Long-input first-audio dropped from ~3 000 ms (v0.7 serial) to 41-52 ms (v1.2 streaming).** 60× headline. Time-to-first-sentence on a 500-word agent monologue is now bounded by the first sentence's synth (one short utterance), not the whole input.
- **Inter-chunk gap median 0.02 ms, max 0.61 ms.** Synth thread is consistently one chunk ahead of the audio thread on a 490-word input; the consumer never waits between chunks. Effectively gapless without the custom miniaudio data source (v1.2.1 path).
- **Total wall time (166 s)** = sum of all 47 chunks' audible duration. Streaming doesn't make the speech go faster; it makes the wait disappear before the first sample.

## Honesty

- "First audio" is captured inside `streamS16leAppend` right after `sound.start()`. Real device-pump first-frame is a few ms later (one device period, ~10 ms). The relative number (v0.7 vs v1.2) is correct; the absolute is a tight lower bound.
- Gap measurements are taken between consecutive `streamS16leAppend` returns in the consumer thread. They reflect the producer-keeps-up condition, not the audio-domain silence between samples. With sub-ms gaps the audio domain doesn't perceive a seam either.
- Bench was run twice for sanity; results landed in the same envelope. No multi-iteration averaging because a single long run takes ~167 s — `--warm` defaults to 1 for long inputs in practice.
